## Global constraints (bind every task)

- **Transparent proxy — never rewrite forwarded bytes.** All protocol parsing is *observational*.
  The passthrough sites (`connection.rs:538-568, 742, 853, 930, 1128, 1229`) forward the raw read
  buffer verbatim; no task may make forwarding depend on parse success except the pre-existing
  `mode_enforcer.validate` rejection path. (Vision §4.3, P2 §2)
- **Fail closed.** When the proxy cannot positively classify a connection as clean, it **pins**.
  Unclassifiable ≠ safe. (P2 §4.1)
- **Observability is best-effort.** Event-logging accuracy (Task 9) must never block/slow/fail a
  query or regress passthrough. (Vision §4.4)
- **Latency budget:** <1ms added/query target; hot-path additions (reassembly, extended tracking)
  must stay cheap. (P5, measured later.)
- Each task ends green: `cargo build`, its new tests, and the existing suite.

## Task 1 — Transparency contract (P2 §4.0)

**Deliverable:** `docs/transparency-contract.md` — a short normative contract defining observable
equivalence: for any client interaction, the backend→client message sequence, final result-set bytes,
and all session-observable state (txn status, session GUCs, prepared statements, temp objects, cursors,
advisory locks, LISTEN registrations) are identical to a direct connection, in every pooling mode;
pooling may change *which* backend serves a client between transactions but never observably. State the
fail-closed rule (unclassifiable → pin) and the passthrough invariant. Link it from `docs/architecture.md`
(and `docs/connection-pooling.md` if apt).

**Steps:** (1) write the contract using the spec §4.0 draft as the base, tightened to what the code
actually guarantees today (note Hybrid's dependence on complete detection). (2) Add a one-line pointer
from `docs/architecture.md`. **Guardrail:** none (doc); it is the spec the Task 2 suite encodes. No code.

## Task 3 — Message reassembly + framing fuzzer (P2 §4.4, §5.2)

**Deliverable:** `extract_messages` reassembles messages spanning multiple `read()`s / exceeding the read
buffer; a framing fuzzer guards it.

- Fix `extractor.rs:333-372` to retain incomplete trailing bytes across calls and prepend them to the next
  call's data before parsing (use/replace the `buffer` field `:20`). **Per-direction:** because one
  extractor instance serves both directions in the managed loop (`connection.rs:602` → `extract_messages`
  at `:646` for client→backend AND `extract_error`/`is_query_complete`/`extract_ready_for_query` at
  `:768/795/827` for backend→client), the reassembly buffer must apply **only** to the client→backend
  `extract_messages` stream. Preferred: give the client→backend direction its own extractor instance (or a
  clearly direction-scoped stateful buffer) so backend framing scans are unaffected. Verify the owned-backend
  path (`connection.rs:1027`) too.
- New `scry-proxy/tests/framing_fuzz.rs` (proptest, no container, mirror `pooling_safety_test.rs`): generate
  a batch of valid Postgres frontend messages, concatenate, then split at **arbitrary** byte boundaries fed
  as a sequence of `extract_messages` calls; assert (a) no panic on any fragmentation or malformed/truncated
  input, and (b) the reassembled message sequence equals the sequence from feeding the whole buffer at once
  (reference framing). Include a message larger than a typical read buffer.

**Steps (TDD):** write the fuzzer + a unit test for a Parse split across two reads → currently drops the
tail (red); implement per-direction buffering (green); keep the CRIT-1 DISCARD-ALL split-buffer and
'Z'-in-parameter regressions green. **Guardrail:** framing fuzzer.

## Task 6 — Prepared-statement / transaction-replay honesty (P2 §4.5)

**Deliverable:** no silent state destruction; the recommended *restrict-by-pinning* resolution.

- **Restrict:** in Transaction mode, **pin** any connection that holds protocol-level (Parse) prepared
  statements or an unnamed statement rather than attempting SQL-`PREPARE` replay. Concretely: make holding a
  protocol-level prepared statement a hard pin (so the replay gate at `connection.rs:876` is not entered for
  it) — the correct-by-construction path. Session-variable-only replay may remain.
- **Eliminate the dangle:** the `clear_all()` at `connection.rs:911` (after the prior `ReadyForQuery` was
  already flushed at `:853`) must not silently drop the client's statements. With restrict-by-pinning the
  I/O-failure replay path should no longer be reachable for prepared-statement state; if any replay remains
  (session vars) and its I/O fails, surface a clean error to the client instead of `clear_all()`-and-dangle,
  or tear down the client connection deterministically — never leave the proxy's view silently diverged from
  the backend. Also reconcile the partial-failure branch (`connection.rs:887-902`) which today only logs.

**Steps (TDD):** integration test — prepare a statement via extended protocol in Transaction mode, force a
backend swap, assert the client either keeps working (pinned, same backend) or receives a clean, typed error
— **never** a silent unprepared-statement failure with no proxy signal (red against current `clear_all`);
implement restrict-by-pinning + remove silent destruction (green). **Guardrail:** dirty-connection + pooling-safety.

## Task 7 — Hot-path robustness (P2 §4.7)

**Deliverable:** no panic-amplifiers on the connection path; bounded state.

- Replace `pool_manager.rs` `expect("connection taken")` (`:104`, `:109`) with a typed error (extend
  `AcquireError` @ `:28-38` or a dedicated `ConnectionTaken` error) returned from `stream()`/`stream_mut()`;
  propagate via the `?`/`anyhow` already used at the call sites in `connection.rs`.
- Make the auth-lock reads poison-safe: at `server.rs:410, 480, 959, 1035` handle a poisoned
  `std::sync::RwLock` (recover the guard, or switch the lock type) so one poisoned lock cannot cascade into
  a panic on every new connection's spawn path (959/1035).
- Enforce the prepared-statement map bound: implement LRU eviction in `add_prepared_statement`
  (`connection_state.rs:164-167`) honoring `max_prepared_statements` (`:63-65`, currently dead). Evicting a
  tracked statement must remain fail-closed (if eviction would lose pin-relevant state, keep the connection
  pinned rather than silently forget).

**Steps (TDD):** unit tests for each — `stream()` after `take_connection` returns the typed error (not
panic); a poisoned auth lock still serves reads; LRU evicts oldest beyond the bound and stays fail-closed.
Implement. **Guardrail:** the three unit tests.
