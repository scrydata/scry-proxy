## Global constraints (bind every task)

- **No surface reports success while doing nothing.** Every `SHOW` returns live state; every control command
  performs its action or returns an honest `ErrorResponse` — never a false `CommandComplete`. (P4 §1, §3)
- **Fail closed.** Non-loopback metrics binds and secret-file misconfig degrade to a startup refusal, not silent
  exposure. (Vision §4.1)
- **Never leak secrets.** `SHOW CONFIG`, config dumps, logs, and metrics never emit `backend.password`,
  `publisher.http_api_key`, `admin.admin_password`, `anonymize_salt`. Reuse `RedactedSecret`. (P4 §4.3, §5.5)
- **Transparent proxy preserved.** Registries/metrics/tracing are observational bookkeeping; they must never
  block, slow materially, or alter the proxied client/backend stream. (Vision §4.3/§4.4)
- **PgBouncer console compatibility.** Keep existing `SHOW` column sets/semantics. (P4 §8)
- Each task ends green: `cargo build`, its new tests, existing suite, and `docs_code_parity_test`.

## Task 1 — AdminHandles plumbing + client/server registries (foundation; P4 §4.1)

**Deliverable:** the shared state the console reads/controls, threaded to `AdminConsole`. No SHOW/command
behavior change yet (those are Tasks 2–5) — this task only builds and wires the infrastructure and proves the
registries populate.

- New `AdminHandles` (Arc-shared, constructed once in `ProxyServer::new`): bundles
  `pool_managers: HashMap<String, Arc<PoolManager>>`, `reload_sender: watch::Sender<()>`, a programmatic
  **shutdown trigger** (a `watch`/`Notify`/`CancellationToken` the `run()` loop also selects on, so SHUTDOWN can
  fire drain — wire the trigger here; Task 5 uses it), `Arc<Config>`, and two new registries.
- `ClientRegistry`: `Arc`-shared map `conn_id → ClientEntry{ addr, user, database, state, connect_time,
  request_time, tls }`. Register at the spawn sites (`server.rs:987`/`:1063`) once identity is known
  (`:1103-1113`); deregister on connection end (extend `ConnectionCountGuard` or the handler's exit). Capture
  the TLS flag currently discarded at `:996`. Keep it cheap (a `Mutex<HashMap>` or `dashmap`-free `RwLock` is
  fine; do not add a new heavy dep).
- `ServerRegistry`: expose per-backend/pool state for `SHOW SERVERS` — at minimum per-pool `PoolStatus` plus
  backend identity; a full per-socket backend registry is nice-to-have but pool-level is acceptable for 1.0.
  State which granularity you implemented.
- Thread `Arc<AdminHandles>` into `handle_admin_connection` (`server.rs:1201`) and change
  `AdminConsole::new(...)` to receive it (keep `metrics`). Do NOT restructure `run()`'s by-value `self` more than
  necessary — prefer cloning Arcs into the spawn closures as `pool_managers` already is (`:975`/`:1052`).

**Steps (TDD):** integration test — open N client connections through the proxy, assert `ClientRegistry` has N
entries with correct addr/user/db, and they deregister on disconnect. Unit-test `AdminHandles` construction.
**Guardrail:** the registry-population test (consumed by Task 2's truthfulness tests).

## Task 4 — Functional PAUSE / RESUME / RELOAD (P4 §4.2, §5.4)

- **PAUSE/RESUME [db]:** add an `accepting` flag (AtomicBool/watch) on `PoolManager`; check it at the top of
  `acquire()` (`pool_manager.rs:198`) and return a new `AcquireError::Paused` when paused (client gets an honest
  error; in-flight txns finish since only new acquisition is gated). PAUSE with a db targets that pool via
  `pool_managers`; bare PAUSE pauses all. RESUME flips it back.
- **RELOAD:** make `apply_config_reload` return `Result<(), _>` (surface the auth_file reload failure it
  currently swallows at `server.rs:509-515`); have the admin `RELOAD` invoke it (via an `AdminHandles` capability)
  and return the real result — `CommandComplete` on success, `ErrorResponse` on failure. Do NOT expand
  hot-reloadable scope (still auth_file); just make it honest. (Optionally also fix the TCP-only-loop reload gap —
  note if deferred.)
- Every command: honest result, no false success.
**Steps (TDD):** command-effect tests — PAUSE then assert a new connection's acquire is rejected/blocked and an
in-flight txn still completes; RESUME restores; RELOAD with a broken auth_file returns an ErrorResponse (RED
against the stubs). **Guardrail:** command-effect suite (`tests/admin_command_effect_test.rs`).

## Task 5 — Functional SHUTDOWN / KILL (P4 §4.2, §5.4)

The heaviest task — adds two control *mechanisms*. If either mechanism balloons, split and report.
- **SHUTDOWN [WAIT]:** wire the programmatic shutdown trigger from Task 1 so the admin command initiates the
  existing drain (`server.rs:906`). Restructure so `drain` is reachable (e.g. `run()` selects on the trigger
  alongside signals). `WAIT` blocks until drained (honor `shutdown_timeout_secs`); no-WAIT returns immediately
  after initiating. Honest error if drain can't start.
- **KILL [db]:** build the `conn_id → AbortHandle` (or per-conn `CancellationToken`) registry (capture from
  `JoinSet::spawn` at `server.rs:987`/`:1063`, or thread a token into `ConnectionHandler`), and terminate the
  tracked client connections for the target db via `ClientRegistry`. Return the count / honest error.
**Steps (TDD):** command-effect tests — KILL disconnects the targeted db's clients (assert their sockets close)
and leaves others; SHUTDOWN drains a proxy with an in-flight connection within the timeout. Use a dedicated test
proxy instance so SHUTDOWN doesn't tear down shared state. **Guardrail:** command-effect suite (extends Task 4's).

## Task 6 — `password_file` generic secret loading (P4 §4.4)

both-set conflict rejected; parity test green. **Guardrail:** config-validation + parity tests.
